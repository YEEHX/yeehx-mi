## v1.9.1 · 2026-07-03（进文件夹不再卡）
- 修复：进入素材多、打标重的文件夹要等好几秒才打开。根因是浏览接口给每个素材单独算一次生效标签，每次都全量加载标签表 + 卷内全部文件夹 + 整张 folder_tags（搜索页 v1.1 就修过的同款 N+1，浏览页漏了）；现改为整层一次批量水合。合成重库实测：300 素材的文件夹 1.7s → 11ms
- 修复：文件夹「共同标签」聚合查询强制按 范围文件夹→素材→生效标签 的索引顺序执行（CROSS JOIN），不再从整张 asset_effective 入手——旧写法每次进文件夹固定白付 200ms+，且库越大越慢。实测叶子文件夹 254ms → 1ms，2 万素材大文件夹 268ms → 73ms

## v1.9.0 · 2026-06-13（同步=双向对账 + 重扫不重打）
- 「同步」升级为双向对账：一次点击 = 新增/改动文件入库 → 缺图的生成缩略图 → 只给未打标/失败的打标 → 已消失文件清库记录（盘在线才清）。硬盘加了新素材后，选中盘或文件夹点「同步」就够了
- 扫描链收尾改为增量：快扫/扫描文件夹对同一范围反复跑，不再重打已打标素材、不再重生成已有缩略图——之前重扫整盘会把所有素材重打一遍，白烧 API 调用。强制重打仍走选中后的「打标」按钮，重出缩略图走「生图」/LUT
- 收尾还会把"有图但没打成标"的素材（手动只生过图、上次打标失败）直接排进打标队列，不再漏掉
- 顺序保证：同步先扫描后清理（同队列串行执行），搬家/改名识别先于清理，移动过的素材标签不丢

## v1.8.5 · 2026-06-12（重置后打标不启动 · 终修）
- 根因：任务泵线程的轮询语句没有任何异常保护，重置数据库存在一个"新库文件已建、表还没建完"的毫秒窗口，泵线程撞上就静默死亡——表现为生图正常、打标永远不来，重启才好
- 修复一：重置改为"新库整个建好后才切换"，并发读永远看到完整结构，窗口归零
- 修复二：泵线程改为打不死——任何异常记日志、歇一拍、继续轮询；worker 的状态轮询同样加固
- 新增两个回归测试：并发读 hammer × 12 次重置零异常；向泵注入 3 次异常后任务必须照常完成

## v1.8.1 · 2026-06-12（真机反馈三修复）
- 修「重置数据库后打标/生图卡死、重启才动」：根因是任务泵线程攥着已删除旧库文件的句柄（幽灵库），重置后新任务它看不见。现在连接按"代数"判废，重置后泵自动连上新库，不用重启
- 本地打标提速：把全程不变的标签库大块挪到 prompt 最前面，吃满 ollama/llama.cpp 前缀缓存——旧版每条素材都把几千 token 的词表重新 prefill 一遍，这就是"平时用模型不慢、打标特别慢"的主因
- 手机连不上自检：设置页现在实测服务是否真的在监听局域网，没监听会黄字提醒"先停止再启动 + macOS 防火墙要点允许"；并提示公司/酒店网的"客户端隔离"连不通属网络问题
